Data Quality Audit Toolkit

Business Problem

市場情報與分析團隊的決策，最終都建立在營運資料之上——但這些資料很少是乾淨的。缺值的顧客 ID、重複的訂單、對不上的商品參照、算錯的銷售金額，都會悄悄扭曲 dashboard 與分析結果。多數團隊要等到 stakeholder 發現「這個數字怪怪的」才會察覺問題。這個專案示範如何在資料進入 dashboard 之前，就用可重複執行的流程攔截這類問題。

Methodology

1. 用固定亂數種子產生 products / orders / order_items / customers / inventory 五張合成資料表，並刻意注入可被偵測到的資料品質問題。

2. 用 Python + Pandas 實作 9 類檢核：缺值、重複、ID 格式、價格異常、數量異常、日期邏輯（出貨日早於下單日）、參照完整性、庫存異常、銷售金額計算（quantity × unit_price 是否等於 item_total）。

3. 每個檢核輸出結構化的 issue log（Issue Type / Affected Table / Number of Records / Business Risk / Suggested Fix）。

4. 計算 Completeness、Uniqueness、Validity、Consistency 四個維度分數，並平均出 Overall Data Quality Score。

5. 用 SQL 撰寫對應的檢核查詢（duplicate / missing value / outlier / referential integrity），示範同一套邏輯如何搬到資料倉儲環境執行。

6. 自動產出商業風格的 Markdown 稽核報告，而不是純技術 log。

Key Findings

Consistency 是最弱的維度（95.8/100）

Observation：order_items 裡最多問題集中在 item_total 與 quantity × unit_price 對不起來，以及 product_id 找不到對應商品。

Recommended Action：優先修復 order_items 的計算欄位驗證與商品參照完整性，因為它直接影響營收與商品層級的分析。

20 種問題類型，涵蓋完整的資料生命週期

從缺值、重複、格式錯誤到跨表參照與計算邏輯，每一種都附上受影響筆數、商業風險與具體修復建議，而不是只丟一份技術錯誤清單。

Overall Data Quality Score 97.3/100

在合成資料中刻意注入約 2–5% 的各類問題後，整體工具仍能準確標記並量化這些問題，證明評分方法在真實情境下具備可操作性。

Business Impact

· Dashboard 準確性：攔截會讓 GMV / 訂單量 KPI 被誇大或低估的重複與計算錯誤。

· 銷售分析可信度：找出讓商品層級營收被低估的參照完整性問題。

· 商品資料維護：把格式錯誤、缺類別等問題整理成可交給商品維運團隊的清單。

· 跨部門溝通：用同一份 issue log 與 Data Quality Score，降低 Sales / Finance / Operations 之間「數字對不起來」的爭論。

E-commerce Operations Automation Pipeline

把每日零散的訂單、商品、庫存、出貨來源檔案，自動化轉換成驗證過的 KPI、營運警示、每日報告 與 Dashboard-ready 資料層，並附上 Airflow 排程範例。

Business Problem

電商團隊常收到分散的訂單、商品、庫存與出貨匯出檔案。人工整理容易延誤決策，也讓重複訂單、無效狀態、缺失商品屬性或延遲出貨等問題悄悄扭曲報表。這個專案回答：雜亂的來源資料，如何變成穩定、可重複、Dashboard-ready 的營運流程？

Methodology

1. 用固定亂數種子產生 orders / order_items / products / inventory / shipments / campaign_calendar 六張來源表，並刻意注入重複訂單、無效狀態、缺失商品欄位與極端交易金額。

2. Pipeline：Raw Data → Clean Data → Validate Data → KPI Output → Alert Report → Dashboard-ready Dataset。

3. 用規則型自動化警示：無效訂單狀態、延遲/缺失出貨、庫存低於回補點、商品 GMV 下滑 ≥35%、缺失商品欄位、極端商品單價、重複訂單 ID。

4. 每個洞察用 Metric → Observation → Possible Cause → Recommended Action 結構，適合作為每日營運會議的檢視素材。

5. 附上簡化版 Airflow DAG 範例，展示排程化落地的方向（本地 MVP 不需要真的安裝 Airflow）。

Key Findings

Insight 2 — 履約健康度

Observation：履約率 96.1%，有 1,018 筆延遲或未出貨的合格訂單。

Recommended Action：依承諾出貨日與客戶影響程度，優先處理高嚴重度延遲出貨佇列，並每日核對缺失的出貨掃描紀錄。

Insight 3 — 庫存風險

Observation：9 個商品觸發低庫存警示。

Recommended Action：優先補貨排名較高的商品，並用前置時間加上近期銷售速度重新設定回補點。

Insight 5 — 資料可靠性

Observation：驗證流程找出 45 筆資料錯誤，並產生共 155 筆品質與營運警示。

Recommended Action：在發布主管 KPI 之前，先隔離無效紀錄、指派警示負責人，並依來源系統追蹤重複發生率。

Business Impact

· 減少每日人工彙整的重複工作。

· 統一 KPI 定義，避免各部門各自定義造成的落差。

· 防止資料品質問題悄悄污染決策。

· 讓履約與庫存例外情況被優先處理。

· 產出可直接餵給 BI dashboard 的 CSV 資料契約。
